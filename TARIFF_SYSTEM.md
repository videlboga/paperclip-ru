# Тарифная система в Paperclip

## Текущее состояние

**СУЩЕСТВУЕТ** - система бюджетов (`budgets.ts`, `costs.ts`), которая ограничивает расходы в центах.

**ОТСУТСТВУЕТ** - система тарифных планов (free/premium/pro) с ограничениями по запросам, токенам, функционалу.

## Бюджетная система (существует)

Файл: `server/src/services/budgets.ts`

BudgetService работает по следующему принципу:
- Проверяет политики по `metric === "billed_cents"` (расходы в центах)
- `warnPercent` — порог предупреждения (например, 80%)
- `hardStopEnabled` + `amount` — жёсткий лимит расходов
- При превышении: ставит `status: "paused"` у агента/компании

## Тарифная sistema (не реализована)

### Проблема (VIDA-7)

Пользователь 648981358 (Telegram) имеет "бесплатный тариф", но он **не ограничен**:
- Пользователь может делать неограниченное количество запросов
- Пользователь может использовать неограниченное количество токенов
- Нет проверки тарифа при создании новых запусков

### Что нужно реализовать

1. **Схема тарифов в DB**
   - Таблица `tariffPlans` (free, premium, pro)
   - Таблица `userTariffs` (user → tariffPlan)
   - Лимиты: запросы/месяц, токены/месяц, признаки функционала

2. **API endpoints**
   - `GET /api/companies/:id/tariff` — текущий тариф
   - `GET /api/companies/:id/tariff/limits` — лимиты
   - `PATCH /api/companies/:id/tariff` — смена тарифа (board/ceo)

3. **Проверка при создании запуска**
   - В `agentService.createInvocation()`
   - Проверка `tariff.limits.requestsLeft > 0`
   - Проверка `tariff.limits.tokensLeft > 0`
   - Возможность `hardStop` или `softWarning`

4. **Мониторинг и логирование**
   - Счётчик запросов/токенов по тарифу
   - Уведомления при приближении к лимиту
   - Логирование превышений

### Варианты реализации

#### Вариант A: Интеграция с существующей системой (budgetService)

Использовать текущую структуру `budgetPolicies`, но расширить `metric`:
- `"billed_cents"` — расходы в центах (существует)
- `"request_count"` — количество запросов (новое)
- `"input_tokens"` — входящие токены (новое)
- `"output_tokens"` — исходящие токены (новое)

**Плюсы:** Используем существующий механизм `budgetService.evaluateCostEvent()`
**Минусы:** Смешиваем бюджеты и тарифы в одной структуре

#### Вариант B: Отдельная тарифная система

Новая структура:
- `tariffPlans` — планы (free, premium, pro)
- `userTariffs` — привязка пользователя к плану
- `tariffUsage` — текущее использование (счётчики)
- `tariffEnforcement` — проверка лимитов перед запуском

**Плюсы:** Чистая separation of concerns
**Минусы:** Нужно дублировать логику `budgetService`

#### Вариант C: External adapter (Hermes/Droid)

Если используется HERMES или Droid адаптер, тариф может храниться в них.

**Проверить:**
- Загружены ли адаптеры: `curl -s "http://127.0.0.1:3100/api/adapters"`
- Есть ли plugins: `curl -s "http://127.0.0.1:3100/api/plugins"`

**Текущее состояние:** Адаптеры builtin (claude_local, codex_local), plugins пусты.

## Рекомендация

1. **Срочно**: Проверить, есть ли тариф в HERMES/Droid адаптерах
2. **Если нет**: Реализовать Вариант B (отдельная тарифная система)
3. **Интеграция**: Встроить проверку тарифа в `agentService.createInvocation()`
